# 周度转日度线性插值教程

> 适用场景：你手里只有**周粒度**数据，但分析、建模或看板需要**日粒度**数据。  
> 参考脚本：`DML-iceberg_zjyprc_hadoop.tmp.warranty_cnt_d.sql`

---

## 1. 先说结论：这套算法到底在干什么？

一句话概括：

> **先拿到“上周末值”和“本周末值”，再把一周展开成多天，然后对不同类型的指标采用不同的分配方式，得到日粒度结果。**

这套方法通常包含 4 步：

1. **取周数据**：每行代表 1 周
2. **找到上周锚点**：比如上周末的 `warranty_cnt`
3. **把一周展开成多天**：生成 `day_idx = 1,2,3,...,7`
4. **按指标类型落到每天**：
   - `warranty_cnt`：线性插值
   - `ord_cnt`：均匀分摊

所以它不是单纯的“数学公式”，而是一个完整的**周转日数据建模套路**。

---

## 2. 为什么会需要这种算法？

现实里经常遇到这种情况：

- 上游系统每周产出 1 条汇总数据
- 但是下游分析需要每天的数据走势
- 又不能等真实日表补齐
- 于是就需要一种“合理估算”的方式

比如：

- 在保量 `warranty_cnt`：周表有，但日报没有
- 工单量 `ord_cnt`：周总量有，但每天的分布没有
- FFR / 故障率：需要 `日工单 / 日在保`

这时你就要把周数据“拆开”成日数据。

注意：

> **这不是还原真实每天发生了什么，而是在已知周数据的前提下，构造一份“分析上可用、业务上合理”的日粒度数据。**

---

## 3. 先建立一个最重要的认知：指标不是一类东西

这是理解整段 SQL 的关键。

### 3.1 存量/状态类指标

例如：

- 在保量 `warranty_cnt`
- 用户保有量
- 库存量
- 累计激活设备数

这类指标有个特点：

> 它描述的是“某个时点的状态”，通常变化是连续的、平滑的。

所以它适合用：

> **线性插值（Linear Interpolation）**

---

### 3.2 流量/发生额类指标

例如：

- 周工单数 `ord_cnt`
- 周销量
- 周点击量
- 周投诉单量

这类指标有个特点：

> 它描述的是“某段时间内发生了多少”，本身就是可加总的。

所以它更适合用：

> **均匀分摊（Average Allocation）**

---

### 3.3 为什么不能所有指标都用一个公式？

因为业务语义不一样。

如果把 `warranty_cnt` 也直接除以 7，会出大问题：

- `warranty_cnt` 是“存量”，不是“本周新增”
- 它本来就不是用来平均分摊的
- 它更像是一条从上周值走到本周值的曲线

反过来，如果把 `ord_cnt` 也拿去做插值，逻辑也不自然：

- `ord_cnt` 是这周累计发生的总量
- 它没有“上周末值 -> 本周末值”的状态过渡意义

所以：

| 指标类型 | 例子 | 适合方法 |
|---|---|---|
| 存量 / 状态类 | `warranty_cnt` | 线性插值 |
| 流量 / 发生额类 | `ord_cnt` | 均匀分摊 |

---

## 4. 用最直观的话理解“线性插值”

假设你知道两件事：

- 上周末在保量：100
- 本周末在保量：170

但你不知道周一到周六每天是多少。

一个最自然的假设就是：

> 这一周是“平滑增长”的。

那这一周总共增长了：

```text
170 - 100 = 70
```

如果这一周有 7 天，那平均每天增长：

```text
70 / 7 = 10
```

于是：

- 第 1 天：110
- 第 2 天：120
- 第 3 天：130
- ...
- 第 7 天：170

这就是线性插值最朴素的直觉：

> **在两个已知点之间，按一条直线把中间的点补出来。**

---

## 5. 数学公式怎么写？

### 5.1 标准线性插值公式

```text
当前值 = 起点值 + (终点值 - 起点值) × 当前进度
```

映射到这个案例里：

- 起点值 = `prev_warranty`
- 终点值 = `warranty_cnt`
- 当前进度 = `day_idx / span`

于是得到：

```text
daily_warranty_cnt
= prev_warranty + (warranty_cnt - prev_warranty) * day_idx / span
```

这正是你 SQL 里的核心公式。

---

### 5.2 均匀分摊公式

如果一周工单总量为 `ord_cnt`，跨度是 `span` 天，那么：

```text
daily_ord_cnt = ord_cnt / span
```

这个公式很简单，本质就是：

> **总量平均分给每一天。**

---

## 6. 你这份 SQL 的整体结构

参考脚本分成 3 层：

### 第一层：`weekly` CTE

作用：

- 从周表 `ffr_w` 取数
- 保留各种产品维度、周度维度
- 通过 `LAG` 取出上周的 `warranty_cnt`

关键代码：

```sql
LAG(warranty_cnt) OVER (
    PARTITION BY model_id
    ORDER BY natural_week
) AS prev_warranty
```

它的含义是：

> 在每个 `model_id` 内，按周排序，取上一周的 `warranty_cnt` 作为本周插值的起点。

这一步非常关键，因为：

> **没有起点，就没法做插值。**

---

### 第二层：`daily` CTE

作用：

- 把每周 1 行展开成多行日数据
- 计算 `span`
- 计算 `cal_date`
- 计算 `daily_warranty_cnt`
- 计算 `daily_ord_cnt`

这层是整份脚本的核心。

---

### 第三层：最终 SELECT

作用：

- 输出最终日表字段
- 计算 `daily_ffr`
- 过滤空值行
- 写入 `${date-1}` 分区

关键逻辑：

```sql
CASE WHEN daily_warranty_cnt > 0
THEN daily_ord_cnt / daily_warranty_cnt
ELSE 0 END AS daily_ffr
```

也就是：

> **先得到每天的分子和分母，再去算每天的率。**

---

## 7. 最难理解的部分：为什么要先“展开天数”？

你的 SQL 里最核心的一段是：

```sql
LATERAL VIEW explode(
    sequence(1, span)
) t AS day_idx
```

你可以把它理解成：

> “原来周表一行只有一个周结果，现在我要人为造出第 1 天、第 2 天、第 3 天……第 N 天。”

### 7.1 `sequence(1, 7)` 是什么？

它会生成一个数组：

```text
[1, 2, 3, 4, 5, 6, 7]
```

### 7.2 `explode(...)` 是什么？

它会把数组拆成多行：

| 原始周记录 | day_idx |
|---|---|
| 第 10 周 | 1 |
| 第 10 周 | 2 |
| 第 10 周 | 3 |
| ... | ... |
| 第 10 周 | 7 |

### 7.3 `LATERAL VIEW` 是什么？

它相当于把“拆出来的多行”挂回原表这一行上。

所以效果就是：

> **一行周数据，被复制成多行日数据，每一行多一个 `day_idx`。**

这一步叫：

- 时间展开
- 行展开
- 数据稠密化 / densification

不同团队叫法略有不同，但本质一样。

---

## 8. `day_idx` 到底是什么？

`day_idx` 就是“本周第几天”。

例如：

| day_idx | 含义 |
|---|---|
| 1 | 周一 |
| 2 | 周二 |
| 3 | 周三 |
| ... | ... |
| 7 | 周日 |

然后通过下面这句得到具体日历日期：

```sql
date_add(w.monday_of_week, day_idx - 1) AS cal_date
```

这句很直观：

- 第 1 天：`monday + 0`
- 第 2 天：`monday + 1`
- 第 3 天：`monday + 2`

这样就把“第几天”变成了“哪一天”。

---

## 9. `span` 为什么有时是 7，有时不是 7？

这部分是实战里最重要的细节之一。

SQL 里写的是：

```sql
CASE
    WHEN to_date(CAST(${date-1} AS STRING), 'yyyyMMdd')
         BETWEEN w.monday_of_week AND w.sunday_of_week
    THEN datediff(
        to_date(CAST(${date-1} AS STRING), 'yyyyMMdd'),
        date_add(w.monday_of_week, -1)
    )
    ELSE 7
END AS span
```

---

### 9.1 历史周

如果 `${date-1}` 不在这周内，说明这周已经结束了。

那就直接：

```text
span = 7
```

---

### 9.2 当前周

如果 `${date-1}` 落在这周里，说明这周还没结束。

例如：

- 周一到周日是一整周
- 但今天跑数时，`${date-1}` 可能只是周三
- 那么这周目前只发生了 3 天

此时就不能强行展开 7 天，否则会把未来几天也造出来。

所以只展开到 `${date-1}` 为止。

---

### 9.3 为什么写成 `datediff(当前日, monday - 1)`？

这是个很巧妙的写法。

如果 `${date-1}` 是周三：

```text
datediff(周三, 上周日) = 3
```

正好就得到：

- 周一 -> 1
- 周二 -> 2
- 周三 -> 3

这个结果刚好能直接拿来：

- 当作 `span`
- 当作 `sequence(1, span)` 的终点

这比 `datediff(周三, 周一) + 1` 更顺手。

---

## 10. 把你的核心 SQL 翻译成人话

你关心的这一段：

```sql
,w.prev_warranty + (w.warranty_cnt - w.prev_warranty) * day_idx / CASE ... END AS daily_warranty_cnt
,w.ord_cnt / CASE ... END AS daily_ord_cnt
FROM weekly w
LATERAL VIEW explode(sequence(1, CASE ... END)) t AS day_idx
```

翻译成人话就是：

1. 先从 `weekly` 表里取出本周记录
2. 判断本周是历史周还是当前周，得到 `span`
3. 生成 `1 ~ span` 的天序号 `day_idx`
4. 每个 `day_idx` 对应一天 `cal_date`
5. 在保量按“上周值 -> 本周值”做线性插值
6. 工单量按“周总量 / span”做均匀分摊

这就是整段逻辑。

## 11. 用一个完整例子推演一遍（完整周）

假设某个机型一周的数据如下：

- `prev_warranty = 100`
- `warranty_cnt = 170`
- `ord_cnt = 70`
- `span = 7`

那么：

### 11.1 先展开天数

```text
day_idx = 1,2,3,4,5,6,7
```

### 11.2 计算 daily_warranty_cnt

公式：

```text
100 + (170 - 100) * day_idx / 7
= 100 + 70 * day_idx / 7
= 100 + 10 * day_idx
```

得到：

| day_idx | daily_warranty_cnt |
|---|---:|
| 1 | 110 |
| 2 | 120 |
| 3 | 130 |
| 4 | 140 |
| 5 | 150 |
| 6 | 160 |
| 7 | 170 |

这里有个非常容易误解的小点：

> 为什么第 1 天不是 100，而是 110？

因为 `100` 表示的是**上周末锚点**，不是“本周周一的日值”；  
而 `day_idx = 1` 表示已经走到了**本周第 1 个日点**，所以结果会前进到 110。  
换句话说，这套写法补的是“周内各日点位”，不是把上周末再重复记一遍。

### 11.3 计算 daily_ord_cnt

```text
70 / 7 = 10
```

所以每天都是：

| day_idx | daily_ord_cnt |
|---|---:|
| 1 | 10 |
| 2 | 10 |
| 3 | 10 |
| 4 | 10 |
| 5 | 10 |
| 6 | 10 |
| 7 | 10 |

### 11.4 计算 daily_ffr

```text
daily_ffr = daily_ord_cnt / daily_warranty_cnt
```

比如：

- 第 1 天：`10 / 110`
- 第 7 天：`10 / 170`

这就得到一条日粒度的率曲线。

## 12. 再看一个例子：当前周只过了 3 天

假设今天跑数时：

- `${date-1}` = 周三
- 当前周周一是 `monday_of_week`

那么：

```text
span = 3
```

于是只会展开：

```text
day_idx = 1, 2, 3
```

### 12.1 为什么这样更合理？

因为周四到周日还没发生。

如果你硬展开 7 天，会产生两个问题：

1. **未来数据泄漏**：把还没发生的日子也算出来
2. **平均数失真**：本来只发生了 3 天，却除以 7

所以当前周一定要只展开到 `${date-1}`。

这是这份 SQL 非常专业的一点。

---

## 13. 为什么 `WHERE w.prev_warranty IS NOT NULL` 很重要？

你的脚本里有一句：

```sql
WHERE w.prev_warranty IS NOT NULL
```

这表示：

> 只有能找到“上一周值”的记录，才允许做插值。

为什么？

因为插值至少要有两个点：

- 起点
- 终点

如果是第一周，没有上一周，那起点就缺失了。

这时常见处理方式有 3 种：

1. **直接过滤掉**（你这份脚本采用的方式）
2. 用本周值当起点
3. 用 0 当起点（通常风险较大）

你这份脚本选择了最稳妥的方式。

## 14. 这类算法的逻辑结构图

你提到想“把整个思考打散，最后得到逻辑结构”，可以用下面这张思维图来记：

```text
周表数据
  └─ 1. 找锚点
       └─ 上周 warranty_cnt = prev_warranty
  └─ 2. 算跨度 span
       ├─ 历史周 = 7
       └─ 当前周 = 已发生天数
  └─ 3. 展开天序号 day_idx
       └─ sequence(1, span) + explode
  └─ 4. 生成 cal_date
       └─ monday_of_week + day_idx - 1
  └─ 5. 按指标类型计算日值
       ├─ warranty_cnt -> 线性插值
       └─ ord_cnt -> 均匀分摊
  └─ 6. 计算 daily_ffr
       └─ daily_ord_cnt / daily_warranty_cnt
```

如果你把这张结构图记住，类似 SQL 基本都能看懂。

## 15. 常见坑（非常重要）

### 坑 1：`ORDER BY natural_week` 可能跨年有风险

如果 `natural_week` 不是连续唯一的周序号，只按它排序可能在跨年时出问题。

更稳妥的方式通常是：

- `ORDER BY natural_year, natural_week`
- 或直接 `ORDER BY sunday_of_week`

如果你的源表 `natural_week` 已经包含年份信息，那就没问题；否则要特别小心。

---

### 坑 2：整数除法导致精度丢失

如果 SQL 引擎把下面这类计算按整数处理：

```sql
(w.warranty_cnt - w.prev_warranty) * day_idx / span
```

可能会被截断。

更稳妥的方式是显式转成 decimal/double，例如：

```sql
CAST(w.prev_warranty AS DOUBLE)
+ (CAST(w.warranty_cnt AS DOUBLE) - CAST(w.prev_warranty AS DOUBLE))
  * day_idx / span
```

如果上游字段本身就是 decimal，问题就小一些，但依然值得确认。

---

### 坑 3：`model_id` 作为分区键是否足够

现在脚本里：

```sql
PARTITION BY model_id
```

如果同一个 `model_id` 会在不同国家、品牌、产品线下重复出现，理论上可能串周。

所以实际要确认：

> `model_id` 是否已经能唯一代表一条连续时间序列？

如果不能，窗口分区键应补充更多维度。

---

### 坑 4：第一周被过滤是正常现象，不要误以为丢数

很多人看到 `prev_warranty IS NOT NULL` 后会以为“怎么少了一周”。

其实不是丢数，而是：

> **第一周没有前置锚点，所以不满足插值条件。**

---

### 坑 5：当前周不能强行按 7 天展开

这是最常见错误之一。

如果当前周只过了 2 天，却按 7 天分摊：

- `daily_ord_cnt` 会偏小
- `daily_warranty_cnt` 会被摊平到未来天数
- 下游日 FFR 会失真

---

## 16. 怎么验证这类插值 SQL 算得对不对？

建议你每次都做下面 5 个检查。

### 检查 1：展开行数是否正确

- 历史周应该展开成 7 行
- 当前周应该展开成已发生天数行

### 检查 2：最后一天的 `daily_warranty_cnt` 是否等于 `warranty_cnt`

因为公式里：

```text
day_idx = span 时
daily_warranty_cnt = prev + (curr - prev) * 1 = curr
```

这是插值正确性的核心校验点。

### 检查 3：第一天是否接近起点后的第一个日点

例如完整 7 天场景下：

```text
day 1 = prev + (curr - prev) / 7
```

### 检查 4：`daily_ord_cnt * span` 是否回到 `ord_cnt`

因为是均匀分摊，所以应满足：

```text
sum(daily_ord_cnt) = ord_cnt
```

### 检查 5：日率是否满足业务常识

比如：

- 分母越大，率通常越小
- 如果在保量递增而工单均摊不变，FFR 应大体往下走

## 17. 这套方法适合哪些业务场景？

常见应用场景：

1. 周度在保量 -> 日度在保量
2. 周度保有量 -> 日度率指标分母
3. 周度库存点位 -> 日度库存估算
4. 周度累计用户数 -> 日度趋势展示
5. 周度汇总表 -> 日粒度看板兜底方案

---

## 18. 什么场景下不适合直接用线性插值？

如果数据明显不是“平滑变化”的，就要谨慎。

例如：

- 周内明显存在节假日峰值
- 周末和工作日差异极大
- 业务有强烈的波峰波谷
- 上游其实已经有真实日数据，只是你没接上

这时线性插值只是一个“近似方案”，不是最好方案。

更高级的方法可能包括：

- 按历史日分布权重分摊
- 按工作日/周末不同权重分配
- 按节假日模型修正
- 直接建设真实日表

所以：

> **线性插值的前提是：你接受“平滑变化”的业务假设。**

## 19. 一个通用复用模板

如果以后你还要写类似 SQL，可以按这个骨架：

```sql
WITH weekly AS (
    SELECT
        dim1,
        dim2,
        monday_of_week,
        sunday_of_week,
        metric_stock,
        metric_flow,
        LAG(metric_stock) OVER (
            PARTITION BY dim1
            ORDER BY sunday_of_week
        ) AS prev_stock
    FROM source_weekly
),
daily AS (
    SELECT
        w.*,
        day_idx,
        date_add(w.monday_of_week, day_idx - 1) AS cal_date,
        CASE
            WHEN ${biz_date} BETWEEN w.monday_of_week AND w.sunday_of_week
            THEN datediff(${biz_date}, date_add(w.monday_of_week, -1))
            ELSE 7
        END AS span,
        w.prev_stock + (w.metric_stock - w.prev_stock) * day_idx /
        CASE
            WHEN ${biz_date} BETWEEN w.monday_of_week AND w.sunday_of_week
            THEN datediff(${biz_date}, date_add(w.monday_of_week, -1))
            ELSE 7
        END AS daily_stock,
        w.metric_flow /
        CASE
            WHEN ${biz_date} BETWEEN w.monday_of_week AND w.sunday_of_week
            THEN datediff(${biz_date}, date_add(w.monday_of_week, -1))
            ELSE 7
        END AS daily_flow
    FROM weekly w
    LATERAL VIEW explode(
        sequence(
            1,
            CASE
                WHEN ${biz_date} BETWEEN w.monday_of_week AND w.sunday_of_week
                THEN CAST(datediff(${biz_date}, date_add(w.monday_of_week, -1)) AS INT)
                ELSE 7
            END
        )
    ) t AS day_idx
    WHERE w.prev_stock IS NOT NULL
)
SELECT *
FROM daily;
```

记忆口诀：

> **先 LAG，再 span；先 explode，再按指标算。**

## 20. 你可以怎么向别人解释这段 SQL？

如果同事问你“这段 SQL 到底在干嘛”，你可以直接这样说：

> 这段 SQL 是把周粒度的在保量和工单量转成日粒度。  
> 它先用 `LAG` 找到上周的在保量作为插值起点，再根据当前周是否结束来算出天数跨度 `span`，然后用 `sequence + explode` 把一周展开成多天。  
> 对于在保量这种存量指标，用线性插值；对于工单量这种流量指标，用均匀分摊；最后再算每天的 FFR。

这就是一段专业但不晦涩的解释。

## 22. 最后做个总复盘

这份脚本里最核心的认知，不是某个 SQL 语法，而是下面这 4 句话：

1. **周表转日表，第一步不是算值，而是先展开行。**
2. **存量指标适合插值，流量指标适合分摊。**

## 23. 一页速记版

### 目标

把周数据转成日数据。

### 核心做法

- `LAG`：取上周值
- `span`：确定这周该展开几天
- `sequence + explode`：生成每天
- 线性插值：给存量指标
- 均匀分摊：给流量指标

### 对应公式

```text
daily_warranty_cnt = prev_warranty + (warranty_cnt - prev_warranty) * day_idx / span
daily_ord_cnt      = ord_cnt / span
daily_ffr          = daily_ord_cnt / daily_warranty_cnt
```

### 核心判断

```text
历史周 -> span = 7
当前周 -> span = 已发生天数
```

### 核心风险

- 窗口排序是否跨年正确
- 除法是否有精度损失
- 分区键是否足够唯一
- 当前周是否误展开到未来
